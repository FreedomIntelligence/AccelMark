# Contributing to AccelMark

## Ways to Contribute

1. **Submit a benchmark result** for a chip not yet in the leaderboard
2. **Add platform support** by writing a new script under `scripts/`
3. **Improve existing scripts** for correctness or clarity
4. **Report a suspicious result** by opening an issue

---

## Submitting a Result

### Requirements

- You must run on one of the defined suites (A, B, C, or D)
- You must use `loadgen/loadgen.py` for request sending (do not implement your own timing logic)
- You must run at least 3 times (`num_runs >= 3`)
- You must pass `scripts/validate_submission.py` before opening a PR

### Step-by-Step

```bash
# 1. Collect environment info (required, do not skip)
python scripts/collect_env.py --output my_submission/env_info.json

# 2. Run the benchmark using your platform's script
#    For a new platform, copy scripts/template/run_benchmark.py
python scripts/{platform}/run_{framework}.py \
  --suite suite_A \
  --scenario offline \
  --output-dir my_submission/

# 3. Run accuracy check (required for all inference suites)
python scripts/run_accuracy.py \
  --suite suite_A \
  --script scripts/{platform}/run_{framework}.py \
  --output my_submission/accuracy.json

# 4. Validate everything
python scripts/validate_submission.py --dir my_submission/
# Must exit 0 before opening PR

# 5. Name your submission directory
# Format: {chip_shortname}_{model_shortname}_{suite_id}_{scenario}_{date}
# Example: h100x1_llama3-8b_suite-A_offline_2026-03-20
mv my_submission results/community/h100x1_llama3-8b_suite-A_offline_2026-03-20

# 6. Open a pull request
```

### Submission Directory Contents

```
results/community/your_submission_name/
├── result.json          ← auto-generated by your benchmark script
├── env_info.json        ← auto-generated by collect_env.py
├── accuracy.json        ← auto-generated by run_accuracy.py
├── run.log              ← terminal output from your benchmark run
├── samples.jsonl        ← optional but strongly recommended
└── run_benchmark.py     ← the exact script you used
```

## Trust Tiers

| Tier | Meaning | Leaderboard |
|------|---------|-------------|
| `verified` | Independently reproduced by a maintainer (within 5%) | Main table |
| `community` | Schema-valid, passes checks, not yet reproduced | Community table |
| `flagged` | Anomaly detected or reproduction failed | Under Review section |

Most results will stay in `community` tier. This is fine.
Community results are still shown on the leaderboard, clearly labeled.

## Adding a New Platform

1. Copy `scripts/template/run_benchmark.py` to `scripts/{platform}/run_{framework}.py`
2. Implement `inference_fn` for your platform
3. Add `scripts/{platform}/README.md` with setup instructions
4. Add `scripts/{platform}/requirements.txt`
5. Submit a result using your new script as proof it works

---

## Challenging a Result

If you believe a submitted result is incorrect:
1. Open a GitHub issue using the "Challenge a Result" template
2. Include: which result, what you observed, evidence (ideally your own run on same hardware)
3. Maintainers will investigate and may move the result to `flagged`

## Using Local Models (Air-gapped / No Internet Access)

AccelMark supports running benchmarks with locally stored model weights.
This is common in air-gapped environments or private clusters.

### How it works

The `model_id` and `model_revision` fields in `result.json` are used for
identification and comparison on the leaderboard — they do not need to match
the actual loading path at runtime.

### Steps

**1. Download the model in advance** (on a machine with internet access):

```bash
pip install huggingface_hub
huggingface-cli download meta-llama/Meta-Llama-3-8B-Instruct \
    --local-dir /your/local/path/Meta-Llama-3-8B-Instruct
```

**2. Run the benchmark pointing to the local path:**

```bash
PYTHONPATH=. python scripts/nvidia/run_vllm.py \
    --suite suite_A \
    --scenario offline \
    --output-dir ./my_submission/ \
    --tensor-parallel-size 1 \
    --model-path /your/local/path/Meta-Llama-3-8B-Instruct
```

**3. Keep the standard model identifier in result.json:**

The script automatically fills `model_id` with the suite's standard HuggingFace
identifier and `model_revision` with the locked hash. Do not change these fields
even when loading from a local path — they exist for leaderboard comparability,
not to describe where the model was loaded from.

**4. If your local copy has a different revision hash:**

HuggingFace repositories sometimes update non-weight files (README, metadata)
which changes the commit hash without changing model weights. If your local copy
was downloaded at a different point in time but has identical weight files
(`*.safetensors`, `config.json`, `tokenizer.json`), add a note in `meta.notes`:

```json
"notes": "Local model copy, weights identical to locked revision 8afb486c"
```

### What counts as the "same model"

| Changed file | Affects inference? | Need to re-download? |
|---|---|---|
| README.md | No | No |
| .gitattributes | No | No |
| config.json | Yes | Yes |
| tokenizer.json | Yes | Yes |
| model-*.safetensors | Yes | Yes |

If only README or metadata changed, your local copy is considered equivalent
to the locked revision for AccelMark purposes.

## Code of Conduct

- Be specific when challenging results — vague accusations are not actionable
- Vendor employees may submit results for their own chips (labeled with "Vendor" badge)
- Do not open PRs that modify `schema/accuracy_subset.jsonl` — this file is immutable
